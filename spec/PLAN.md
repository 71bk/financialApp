# financialApp 專案計畫書

> 最後更新：2026-07-07

## 一、產品定位

**「訂閱與分期的未來現金流規劃器」**——不是又一個記帳 App。

台灣十幾款記帳 App 都在比「怎麼記錄過去」，本產品的主軸是「**發薪日那一刻就告訴你：這個月的錢會怎麼被吃掉、今天還能花多少**」。記帳只是餵資料的手段，規劃才是產品。

- 目標族群：有訂閱、有分期、想知道「我到底還能花多少」的年輕上班族
- 一句話賣點：AI 財務教練，把你的固定收支變成看得懂的未來

## 二、核心概念：固定收支管理

使用者設定一次，之後每月自動套用。這是整個產品的資料底座。

### 固定收入

| 類型 | 例子 | 備註 |
|---|---|---|
| 薪資 | 月薪 | 含發薪日，Safe-to-Spend 以此為週期起點 |
| 定期獎金 | 三節、年終 | 年度週期，攤入現金流投影 |
| 兼職 / 其他 | 家教、租金收入、利息 | 支援多筆收入來源 |

### 固定支出

| 類型 | 例子 | 金額性質 |
|---|---|---|
| 帳單類 | 電話費、網路費、水電瓦斯、保險費 | 電話/網路多為固定；水電為**變動金額（記估計值，帳單到後校正）** |
| 居住 | 房租、管理費、停車費 | 固定 |
| 訂閱 | Netflix、Spotify、iCloud… | 固定，含**免費試用到期追蹤** |
| 分期付款 | 手機、家電、無卡分期 | 含期數、剩餘期數、利率/手續費 →**顯示總利息成本** |

設計要點：

- 每筆固定收支都有：名稱、方向（收入/支出）、金額（固定 or 估計）、週期（月/年/自訂）、日期、分類、付款方式。
- 帳單提醒分兩種：**「即將自動扣款」通知**（訂閱、代扣）與 **「記得去繳費」提醒**（需超商/轉帳手動繳的水電、電話費，以繳費期限為準）。
- 變動金額帳單（水電）先用估計值投影，實際帳單入帳後回填校正；後期可讓 AI 學習季節性模式（如夏季電費）提高投影準確度。

## 三、功能藍圖

### Phase 1 — MVP（目標：能自用、能給朋友用）

| # | 功能 | 備註 |
|---|---|---|
| 1 | Onboarding 即 aha moment | 收入/職業 → 訂閱 → 分期 → 房租/帳單，3 分鐘後揭露「你的真實可支配收入」 |
| 2 | 固定收支管理 | 如第二節；產品資料底座 |
| 3 | 未來 3–6 個月現金流投影 | 差異化核心，最先做 |
| 4 | Safe-to-Spend | 首頁大數字：「今天還能花 NT$XXX」＝（定期收入 − 固定支出 − 分期 − 儲蓄目標）÷ 當期剩餘天數 |
| 5 | 扣款/繳費前推播提醒 | 「Netflix 明天扣 390」「電話費 7/10 前要繳」，最強留存鉤子 |
| 6 | 自然語言快速記帳 | 打「午餐 120」→ LLM 解析入帳；系統鍵盤聽寫免費附贈語音 |
| 7 | 月度分類分析 | 標配，圓餅＋趨勢即可，不花創意 |
| 8 | AI 每月預算建議 | 用 LLM API，根據收入/職業/固定收支生成 |
| 9 | 備份與匯出 | OS 自動備份（Auto Backup／iCloud）涵蓋資料庫；JSON 完整備份檔匯出／匯入（跨平台換機用）＋CSV 匯出（人讀分析用）；設定頁顯示備份狀態（ADR-005） |

### Phase 2 — 降低記帳摩擦（解決「載具族很懶」）

策略：**接「錢離開帳戶的那一刻」，不執著於發票。** 三層輸入：自動 → 半自動 → 手動。

1. **Android 通知監聽**：自動解析銀行/LINE Pay/街口扣款推播，一鍵確認入帳
2. **iOS 捷徑整合**：Apple Pay 交易觸發自動記帳＋Siri 快速記帳（同一套基建）
3. **CSV 匯入**：財政部電子發票平台明細、信用卡帳單 → AI 自動分類＋去重
4. **發票 QR Code 掃描**：離線解碼，補實體消費場景（不需任何 API 申請）
5. **Wrapped 式月報**：可分享的 AI 月度總結圖卡（留存＋自然行銷）
6. **儲蓄目標**：緊急預備金/旅遊基金，AI 反推每月存款額
7. **AI 帳單估算**：學習水電等變動帳單的季節性，提高現金流投影準確度
8. **次數型通行證追蹤**：健身房季票、「90 天 3000 元可用 30 次」這類票——打卡記次、每次攤提成本、「還剩 X 次、Y 天」到期提醒（欄位預留見 spec/03-data-model.md）

### Phase 3 — 產品驗證成功後

- 購買決策模擬器（「買 4 萬的東西分 12 期，未來半年現金流長怎樣」）
- 同職業/收入級距的支出結構對比（注意隱私設計）
- 訂閱閒置偵測、漲價追蹤（次數票的低使用率偵測同族）
- 攤提預留視角（sinking fund）：季繳/年繳/次數票攤成每月預留，繳費月不突刺——計算與顯示層功能，不改資料模型
- **成立公司 → ISO 27001 認證 → 申請財政部載具 API**（在此之前一律不碰）

### 明確不做

銀行帳戶整合（麻布的護城河）、共享帳本、投資追蹤、多幣別、遊戲化養成、爬財政部平台（個資法風險）、專屬語音介面、**國際化／多語系**（差異化全依賴台灣在地資料來源，出海等於放掉護城河去打 Copilot/YNAB；重啟條件：台灣 D30 留存驗證通過後另立產品決策。2026-07-07 決定）。

### 載具 API 現況（2026-07 查證）

- 財政部「電子發票應用 API」存在，可用手機條碼查載具發票，但**2023 年起停收個人申請，且開發者須通過 ISO 27001/CNS 27001 認證**。
- MVP 替代路線：發票 QR Code 離線解碼（免申請）＋ 財政部平台 CSV 匯入 ＋ 通知監聽。
- 禁止事項：不得代管使用者手機條碼＋驗證碼去爬財政部平台（個資法風險）。

## 四、技術方向（2026-07-07 定案，決策細節見 `spec/adr/`）

- **App**：**Flutter**（ADR-001）——通知監聽、推播、相機都需要原生能力，PWA 出局。**Android 優先上線**（通知監聽只有 Android 能做，自動記帳體驗最完整），iOS 隨後跟上並以捷徑補位。資料採 local-first SQLite、MVP 無帳號（ADR-002），狀態管理用 Riverpod（ADR-003），AI 經 thin proxy 呼叫（ADR-004）。
- **後端**：輕量起步（Supabase / Firebase 級別）。
- **核心資料模型**：
  - `Profile`：職業、收入級距
  - `RecurringItem`（週期性收支）：方向（收入/支出）、金額類型（固定/估計）、金額、週期、日期/繳費期限、分類、付款方式、提醒類型（自動扣款/手動繳費）；分期擴充欄位（總期數、剩餘期數、利率/手續費）；訂閱擴充欄位（試用期迄日）
  - `Transaction`：日常變動收支（手動/自動來源、去重指紋）
  - `Category`、`SavingGoal`
- **AI**：Claude API——NL 記帳解析、CSV 分類用便宜快的模型（Haiku 4.5），月度分析與預算建議用 Sonnet 5。每使用者每月成本約幾塊台幣，可控。
- **推播排程**：扣款/繳費提醒是留存命脈，第一天就把排程架構做對。
- **隱私**：財務資料敏感，資料最少化、提供匯出功能給信任感，AI 呼叫不留存原始明細。

## 五、風險與現實

1. **紅海市場**：功能面清單 ≈ 現有 App 的聯集，勝負在定位與留存，不在功能數量。
2. **留存是主戰場**：記帳 App 死於三分鐘熱度。扣款提醒、Safe-to-Spend、月報都是為此設計。
3. **iOS 自動記帳先天受限**：只能靠捷徑，體驗會比 Android 差，行銷上要誠實。
4. **可被模仿**：麻布/MOZE 若跟進現金流規劃，他們有資料優勢。窗口是速度和專注。
5. **商業模式**：免費＋訂閱制進階（AI 教練、模擬器放 premium）。先驗證留存，再想營收。

## 六、執行順序與驗證

1. **第 1 步**：定技術棧 → 資料模型 → Onboarding + 固定收支管理 + Safe-to-Spend（先做出打開就有感的殼）
2. **第 2 步**：推播提醒 + NL 記帳 + 現金流投影 → **自己用一個月**（dogfooding）
3. **第 3 步**：AI 建議 + 分析圖表 → 給 10–20 個朋友 beta
4. **驗證指標**：D30 留存、每週主動記帳次數、扣款提醒帶回率
5. 指標及格 → 上架 + Phase 2；不及格 → 回頭修定位，不是加功能
